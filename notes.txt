     treats BLANK() as epoch-zero, so BLANK() <= EndDate is TRUE in DAX.
   - Fixed both: Strategic Pool now = 2, Closing Headcount now = 47,
     Attrition % now = 9.62% (was 10.0%) — all now match reference.

   - Fixed to read /api/v1/utilization/summary's total_employees field (46).

6. Utilization Home weekly trend chart — "Unknown" label bug — RESOLVED
   - A record with null week_start was bucketed into a phantom "Unknown"

BACKEND PERFORMANCE OPTIMIZATION — evidence-based, measured
   Profiled BEFORE optimizing (100 concurrent requests):
   - /roster/skills: 514ms wall time
   - /utilization/records (limit=500): 4093ms wall time (worst)
   - /roster/employees: 453ms
   - /utilization/overview: 471ms

   REAL BUG FOUND: a genuine race condition in data_loader.py — two
   concurrent requests hitting a cold cache could both trigger a
   duplicate Excel re-read. Fixed with a threading.Lock.

   FIX: added a cache_on_df decorator (~40 functions across roster/
   booking/utilization_metrics) caching aggregation results keyed on
   DataFrame identity. TRICKY BUG CAUGHT DURING TESTING: naive id(df)
   keying broke 4 tests because Python can recycle a garbage-collected
   object's memory address for a new unrelated DataFrame — fixed by
   storing a strong reference alongside each cache entry and verifying
   identity on lookup.

   BIGGEST WIN: rewrote records_to_dicts from .iterrows() (measured as
   ~25ms of the ~41ms response time — the single biggest hotspot) to
   vectorized column-wise operations. /utilization/records went from
   44.0ms mean to 17.5ms mean per request.

   Also added: GZipMiddleware for large JSON payloads. Reviewed
   SQLAlchemy/SQLite connection settings — already correct, no change
   needed (SQLite is auth-only, not the main data path).

   AFTER optimization (same 100-concurrent test):
   - /roster/skills: 514ms -> 186ms (2.8x, 195->537 req/s)
   - /utilization/records: 4093ms -> 1198ms (3.4x, biggest win)
   - /roster/employees: 453ms -> 390ms
   - /utilization/overview: 471ms -> 403ms
   Mixed 100-concurrent-user simulation across all 4 endpoints: 459ms
   total wall time.

   REDIS DECISION: evaluated, NOT added. Reasoning: single uvicorn
   process, no multi-worker/Docker deployment yet (Phase 7, still
   local-dev), ~1600 total rows of cacheable data. In-process caching
   gives sub-millisecond hits with zero network/serialization
   overhead; Redis would add real operational complexity (separate
   service to run/monitor/secure) for no measurable benefit at this
   scale. Revisit only if a future phase introduces multi-worker/
   multi-container deployment.

   144/144 tests pass after all changes.

MOBILE RESPONSIVENESS — round 2, real bugs found after "verified" round 1
   FINDING: user found 2 real bugs via actual mobile screenshots that
   the earlier automated scrollWidth-based verification completely
   missed — a lesson that automated width checks aren't sufficient,
   need real visual screenshot review too.
   FINDING 1 (filter overflow): NOT actually on HR Analytics as
   initially suspected — the real bug was on Utilization Home, where
   a flex-wrap layout with min-w-0 let flex-shrink win over wrapping,
   compressing all 4 filters onto one illegible truncated line instead
   of stacking on mobile.
   FIX: switched to a grid layout that properly stacks one filter per
   row below the sm breakpoint.
   FINDING 2 (table scroll affordance, much broader than reported):
   a genuine copy-paste bug affecting 6 table instances across 5 pages
   (Employee Directory, HR Analytics, Results, Employee Utilization,
   Project Utilization x2). The "scroll for more" fade indicator was
   nested INSIDE the scrolling container itself, so it was anchored to
   the end of the scrolled content, not the visible screen edge —
   invisible until the user had already scrolled all the way through.
   Tables were actually functionally scrollable the whole time, just
   had no visible hint that they were.
   FIX: built a shared TableScrollContainer component (fade as a
   sibling of the scroll container, tracks real scroll position via
   listener + ResizeObserver) and applied it to all 6 instances.
   Full 390px screenshot sweep across every page (not just automated
   checks) found nothing else. tsc clean, zero console errors.

DOCKER DATA-FILE MOUNTING FIX
   FINDING: the 3 source Excel files were baked into the Docker image
   at build time (plain COPY), not live-mounted. Updating the files on
   disk had zero effect on a running container — would have silently
   served stale data indefinitely unless a full image rebuild was run.
   FIX: added a live bind-mount (./backend/data:/app/data) in
   docker-compose.yml. Now dropping updated files into backend/data/
   takes effect after `docker compose restart backend` — no rebuild
   needed. Documented clearly in DOCKER.md's new "Updating the source
   Excel data files" section (exact filenames required, restart not
   rebuild, re-run test suite after updating since some tests pin
   exact values against the real files).
