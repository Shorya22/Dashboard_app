=== DATA / BACKEND FIXES ===

1. Amit Singh vs Ankit Singh — RESOLVED
   - Booking sheet has "Amit Singh", ground-truth sheet has "Ankit Singh".
   - Confirmed same person (typo in ground-truth sheet): full name "AMIT
     KUMAR SINGH" in roster. 2 of 3 sources agree on "Amit". Hours/utilization
     reconcile exactly across both spellings for the same 4 weeks.
   - Reference PDF's "Ankit Singh 100%" is not independent proof — it's
     generated from the same sheet carrying the typo.
   - Canonical name = "Amit Singh", documented in known-name-variants.md.

2. Departments count = 30 vs reference 29 — RESOLVED
   - Root cause: casing duplicate in Designation column —
     "SalesForce Core Developer" vs "Salesforce Core Developer" (capital F).
   - Same bug class as already-fixed Seniorirty Level casing duplicates
     (Standard Senior vs Standard senior, etc).
   - Fix: normalize Designation casing before counting (title-case, same
     pattern as seniority normalization). Backend now returns 29.
   - Added designation_casing_mismatch data-quality warning.
   - Frontend bug found after backend fix: 3 pages (HR Home, Workforce,
     Skills & Experience) compute Departments client-side for filter
     support and were NOT using the normalized count — still showed 30.
     Fixed with shared distinctDepartmentsCount() helper in
     employee-filters.ts. All 3 pages now correctly show 29.

3. Strategic Pool showed 0, reference shows 2 — RESOLVED
   - Root cause: get_strategic_pool was checking the raw DOJ (DEPT) string
     column ("TBD") instead of the parsed date column — never matched.
   - Separately: Closing Headcount showed 45 vs reference 47 — root cause
     was pandas NaT always evaluating False on comparisons, but real DAX
     treats BLANK() as epoch-zero, so BLANK() <= EndDate is TRUE in DAX.
   - Fixed both: Strategic Pool now = 2, Closing Headcount now = 47,
     Attrition % now = 9.62% (was 10.0%) — all now match reference.

4. Utilization data — was blocked, now UNBLOCKED
   - Booking sheet updated: grew from 258 rows/2 weeks to 1523 rows/7 weeks
     (2026-04-13 to 2026-05-25) — now overlaps ground-truth sheet's weeks.
   - Ran reconciliation on 152 real employee/week combos: Formula A
     (Client Hours / actual logged weekly total) confirmed correct at
     94.1% exact match. Formula B (fixed 45hr denominator) ruled out.
   - Utilization % endpoints now built and live, values match reference
     (71.43% avg period utilization, 69.09% latest week).

5. Utilization Home "Total Employees" showed 47 instead of 46 — RESOLVED
   - Was reading roster's Active Employees count instead of the booking
     sheet's actual distinct employee count.
   - Fixed to read /api/v1/utilization/summary's total_employees field (46).

6. Utilization Home weekly trend chart — "Unknown" label bug — RESOLVED
   - A record with null week_start was bucketed into a phantom "Unknown"
     group, pushing a real week off the visible chart.
   - Fixed: records with no week_start are now skipped in bucketing.
     All 7 weeks now render correctly.

7. File path bug — RESOLVED
   - Source Excel files were renamed on disk (underscores -> spaces/parens)
     after user updated them. Backend path constants were stale, breaking
     data loading entirely. Fixed both roster and booking file paths.
